Enfoque de la Arquitectura

La arquitectura seleccionada para el sistema del Hotel Cacique Bolívar se basa en el patrón Modelo-Vista-Controlador (MVC), implementado con Spring Boot y Java. Este enfoque permite separar de forma clara la lógica de negocio, la interfaz de usuario y el control de flujo, lo que facilita el mantenimiento y la escalabilidad del proyecto.

Elección de Estilo

El estilo arquitectónico elegido es Arquitectura en Capas bajo el patrón MVC. En este esquema:

  • Modelo: gestiona los datos y reglas de negocio (reservas, clientes, paquetes, pagos).
  • Vista: ofrece una interfaz web accesible desde dispositivos móviles y de escritorio.
  • Controlador: actúa como intermediario entre la vista y el modelo, procesando solicitudes y devolviendo respuestas.

Justificación

La elección del patrón MVC con Spring Boot responde a varios factores clave:

  • Mantenibilidad: la separación de responsabilidades reduce la complejidad y permite modificar módulos de forma independiente.
  • Escalabilidad: el sistema puede crecer incorporando nuevas funcionalidades (paquetes turísticos, PQRS+) sin afectar los módulos existentes.
  • Compatibilidad: al ser una aplicación web, garantiza acceso desde cualquier dispositivo con conexión a Internet.
  • Productividad: Spring Boot proporciona herramientas y configuraciones que agilizan el desarrollo y la integración con bases de datos.

Decisión Arquitectónica 1: Seguridad en el acceso a la BD

Contexto:

El sistema manejará datos sensibles de huéspedes, reservas, pagos y PQRS+, por lo que es necesario proteger la información frente a accesos no autorizados.

Decisión:

Solo la capa de negocio (servidor de aplicaciones Java) tendrá acceso directo a la base de datos MySQL, utilizando credenciales encriptadas y evitando que el cliente web acceda directamente.

Consecuencias:

✅ Positivo: Incrementa la seguridad y evita fugas de información.
❌ Negativo: Puede dificultar pruebas rápidas desde el cliente si no hay acceso directo a BD.



Decisión Arquitectónica 2: Autenticación y manejo de usuarios

Contexto:

Se identificó la necesidad de diferenciar roles (cliente, recepcionista, gerente, administrador) para limitar accesos según el perfil.

Decisión:

Implementar un sistema de autenticación con roles y permisos, administrado desde el módulo de perfiles.

Consecuencias:

✅ Positivo: Previene que un usuario sin permisos acceda a funciones críticas (ej. modificar reservas, anular pagos).
❌ Negativo: Aumenta la complejidad del desarrollo y mantenimiento de roles.



Decisión Arquitectónica 3: Rendimiento en consultas e informes

Contexto:

El sistema debe soportar consultas frecuentes (estado de reservas, disponibilidad de habitaciones, informes de pagos y clientes) con tiempos de respuesta aceptables.

Decisión:

Optimizar la base de datos con índices en campos de búsqueda comunes (código de reserva, número de habitación, documento de cliente, fechas).

Consecuencias:

✅ Positivo: Mejora la velocidad de búsqueda y reduce tiempos de espera.
❌ Negativo: Mayor consumo de espacio en disco y mantenimiento de índices adicionales.



Decisión Arquitectónica 4: Escalabilidad con microservicio de IA

Contexto:

Se plantea que en el futuro se pueda integrar un servicio de IA para análisis de datos (ejemplo: predicciones de ocupación o sugerencias de paquetes turísticos).

Decisión:

Desplegar la IA como microservicio independiente que se comunique vía API con el servidor de aplicaciones.

Consecuencias:

✅ Positivo: Escalable, se puede mejorar la IA sin afectar el resto del sistema.
❌ Negativo: Incrementa la complejidad del despliegue y requiere asegurar comunicación entre servicios.



Decisión Arquitectónica 5: Disponibilidad y copias de seguridad

Contexto:

El sistema debe estar disponible 24/7 para que clientes y recepcionistas puedan consultar reservas en cualquier momento. Además, se deben proteger los datos contra pérdidas accidentales o ataques.

Decisión:

Configurar el sistema con disponibilidad continua y realizar copias de seguridad automáticas cada 6 horas en un servidor externo.

Consecuencias:

✅ Positivo: Se garantiza continuidad del servicio y recuperación de información en caso de fallos.
❌ Negativo: Aumenta el costo de almacenamiento y administración de backups externos.



Decisión Arquitectónica 6: Interfaz de usuario web responsiva

Contexto:

El hotel tiene clientes que acceden desde diferentes dispositivos (PC, tablet, smartphone), por lo que la experiencia debe ser fluida y adaptable.

Decisión:

Diseñar la interfaz en formato web responsivo con HTML5, CSS3 y frameworks adaptativos, usando una paleta definida (gris claro, azul claro y blanco).

Consecuencias:

✅ Positivo: Mejora la usabilidad y accesibilidad para cualquier dispositivo.
❌ Negativo: Aumenta el tiempo de desarrollo y pruebas para garantizar compatibilidad en múltiples navegadores.



Decisión Arquitectónica 7: Tecnología de base de datos relacional (MySQL)

Contexto:

El sistema requiere almacenar reservas, clientes, habitaciones, pagos, paquetes turísticos y PQRS+ con integridad y consistencia de la información.

Decisión:

Usar MySQL como gestor de base de datos relacional, con soporte a transacciones y alto nivel de compatibilidad con Java.

Consecuencias:

✅ Positivo: Garantiza integridad de datos, amplia documentación y soporte; fácil integración con aplicaciones Java.
❌ Negativo: Puede limitar la escalabilidad horizontal frente a bases NoSQL en escenarios de gran volumen.



Decisión Arquitectónica 8: Comunicación segura mediante HTTPS

Contexto:

El sistema estará expuesto en la web para clientes y administradores, por lo que es fundamental garantizar la confidencialidad de datos sensibles como contraseñas, reservas y pagos.

Decisión:

Implementar comunicación cifrada usando HTTPS con TLS, asegurando autenticidad del servidor y confidencialidad en el intercambio de información.

Consecuencias:

✅ Positivo: Protege datos sensibles en tránsito y genera confianza en los usuarios.
❌ Negativo: Aumenta el consumo de recursos del servidor debido al cifrado y requiere certificados válidos renovados periódicamente.



Decisión Arquitectónica 9: Manejo de errores, logs y alertas

Contexto:

El sistema debe garantizar confiabilidad y trazabilidad en las operaciones críticas como reservas, pagos y gestión de clientes. Si ocurre un error (caída del servidor, fallo en conexión BD, datos inválidos, etc.), se debe detectar y notificar de forma clara al usuario y al administrador.

Decisión:

Implementar un módulo de manejo de errores centralizado con tres niveles:

  • Logs internos: registro detallado de errores en servidor de aplicaciones (Java), con nivel de severidad (INFO, WARNING, ERROR, CRITICAL).
  • Alertas al usuario final: mensajes claros en la interfaz (“No se pudo procesar su pago, intente nuevamente o contacte soporte”), evitando mostrar detalles técnicos.
  • Alertas al administrador: notificación automática por correo o panel interno cuando se produzcan errores críticos (ejemplo: falla en la base de datos, caída del sistema de reservas).

    • Consecuencias:

    ✅ Positivo: Mejora la confiabilidad, facilita el diagnóstico de problemas y permite actuar de forma proactiva ante fallas.
    ❌ Negativo: Requiere mayor esfuerzo en el desarrollo y almacenamiento de registros, además de una política de limpieza de logs para evitar saturación.



    Decisión Arquitectónica 10: Consumo de APIs externas

    Contexto:

    El sistema del Hotel Cacique Bolívar debe ofrecer valor agregado a sus clientes, como mostrar disponibilidad de destinos turísticos, integración con pasarelas de pago o incluso conectarse con servicios de mapas (ej. Google Maps) para mejorar la experiencia. Estas integraciones requieren consumir APIs externas de manera segura y eficiente.

    Decisión:

    Implementar un módulo de integración de APIs externas con las siguientes características:

  • Uso de conexiones HTTPSpara proteger la comunicación.
  • Manejo de tokens o llaves de acceso segurasalmacenadas en el servidor (no en el cliente).
  • Establecer tiempos de espera (timeouts)y reintentos limitados para evitar bloqueos en caso de fallo externo.
  • Normalización de las respuestas de las APIs para integrarlas al sistema interno (ej. conversión de datos de reservas turísticas o pagos al formato interno).

    • Consecuencias:

    ✅ Positivo: Amplía la funcionalidad del sistema (ej. pagos en línea, integración con mapas, servicios turísticos locales).
    ❌ Negativo: Dependencia de terceros; si la API externa falla o cambia, puede afectar la operación del sistema.



    Decisión Arquitectónica 11: Migración y cambio de tecnologías

    Contexto:

    La tecnología evoluciona rápidamente y el sistema del Hotel Cacique Bolívar debe adaptarse para mantenerse vigente. Actualmente se plantea el uso de Java y MySQL, pero en el futuro podría ser necesario migrar a otros lenguajes, frameworks o motores de base de datos que ofrezcan mejor rendimiento, escalabilidad o soporte.

    Decisión:

    Diseñar el sistema con una arquitectura modular y desacoplada, que permita reemplazar componentes tecnológicos sin afectar el resto del sistema. Esto incluye:

  • Uso de capas bien definidas (presentación, negocio, datos).
  • Aplicacion de interfaces y controladorespara que el cambio de BD (ej. de MySQL a PostgreSQL) o backend (ej. de Java a .NET o Node.js) no afecte la lógica de negocio.
  • Documentación clara de dependencias y procesos de migración.

    • Consecuencias:

    ✅ Positivo: Permite actualizar el sistema a nuevas tecnologías sin rehacer todo el proyecto, alargando su vida útil.
    ❌ Negativo: Requiere mayor esfuerzo inicial en la planificación y pruebas de integración para asegurar compatibilidad futura.



    Decisión Arquitectónica 12: Soporte de idioma y búsquedas avanzadas

    Contexto:

    El hotel atiende tanto a clientes locales como extranjeros, lo que implica la necesidad de un sistema multilingüe (mínimo español e inglés). Además, los usuarios requieren encontrar rápidamente información sobre reservas, habitaciones, paquetes turísticos o PQRS+ mediante un buscador eficiente.

    Decisión:

    Implementar un módulo de internacionalización (i18n) para la interfaz de usuario, permitiendo cambiar de idioma dinámicamente. Asimismo, integrar un motor de búsqueda optimizado (ej. consultas con filtros por nombre, fechas, tipo de habitación, destino turístico).

    Consecuencias:

    ✅ Positivo: Mejora la experiencia del usuario al adaptarse a diferentes idiomas y facilita la localización rápida de información.
    ❌ Negativo: Incrementa la complejidad del desarrollo y mantenimiento (traducciones, validación de textos y configuración de búsquedas eficientes).



    Decisión Arquitectónica 13: Generación de reportes

    Contexto:

    Los gerentes y administradores del hotel necesitan consultar y exportar información de reservas, pagos, clientes y PQRS+ de forma clara y portable. El sistema debe permitir generar reportes en diferentes formatos para análisis y presentación, evitando la manipulación manual de datos.

    Decisión:

    Incorporar un módulo de generación de reportes con soporte a formatos Excel (XLSX) y PDF. Los reportes podrán filtrarse por rangos de fechas, estados y otros criterios, y estarán disponibles para descarga desde la interfaz web. Se utilizarán librerías estándar (ej. Apache POI o JasperReports) para asegurar compatibilidad.

    Consecuencias:

    ✅ Positivo: Facilita la toma de decisiones y el análisis externo de la información; los reportes pueden compartirse fácilmente con terceros.
    ❌ Negativo: Aumenta el consumo de recursos del servidor durante la generación de reportes grandes y requiere pruebas adicionales para mantener el formato correcto.



    Decisión Arquitectónica 14: Soporte para grandes volúmenes de datos y usuarios

    Contexto:

    El sistema del Hotel Cacique Bolívar debe estar preparado para crecer y manejar desde pequeños volúmenes (100 registros/usuarios) hasta grandes escalas (1.000.000 o más), garantizando estabilidad en la gestión de clientes, reservas, habitaciones, pagos y PQRS+.

    Decisión:

    Adoptar una arquitectura escalable con las siguientes prácticas:

  • Uso de índices y particionamiento en la base de datos para optimizar consultas masivas.
  • Implementación de caché en operaciones frecuentes (ej. disponibilidad de habitaciones).
  • Configuración de balanceadores de carga y posibilidad de escalado horizontal en el servidor de aplicaciones.
  • Planificación de migración a bases de datos distribuidas si la demanda supera la capacidad de un solo servidor.

    • Consecuencias:

    ✅ Positivo: El sistema podrá atender tanto a un hotel pequeño como a una cadena de hoteles con millones de registros, sin rehacer la arquitectura.
    ❌ Negativo: Implica mayor complejidad técnica y costos de infraestructura (servidores adicionales, monitoreo y mantenimiento de escalabilidad).



    Decisión Arquitectónica 15: Integración con redes sociales

    Contexto:

    Los clientes del hotel suelen interactuar en plataformas como Facebook, Instagram y WhatsApp, donde buscan información, promociones o realizan consultas rápidas. Para mejorar la experiencia del cliente y aumentar la visibilidad del hotel, se debe integrar el sistema con estas redes sociales.

    Decisión:

    Implementar un módulo de integración con redes sociales que permita:

  • Iniciar sesión con cuentas de redes sociales (ej. Login with Facebook/Google).
  • Compartir promociones o paquetes turísticos directamente desde el sistema a redes sociales.
  • Conectar el módulo de PQRS+ con canales de mensajería (ej. WhatsApp Business API) para recibir solicitudes o reclamos.

    • Consecuencias:

    ✅ Positivo: Mejora la comunicación con los clientes, aumenta la difusión de promociones y agiliza el servicio al cliente.
    ❌ Negativo: Dependencia de APIs externas de redes sociales, que pueden cambiar políticas, generar costos adicionales o limitar funcionalidades en el futuro.



    Decisión Arquitectónica 16: Cumplimiento de regulaciones y protección de datos

    Contexto:

    El sistema gestionará información sensible de clientes como nombres, documentos de identidad, datos de contacto, historial de reservas y pagos. En Colombia (y a nivel internacional) existen leyes como Habeas Data y GDPR que obligan a proteger la información personal y garantizar los derechos de los usuarios.

    Decisión:

    Adoptar un módulo de cumplimiento legal y protección de datos con las siguientes medidas:

  • Encriptación de datos sensibles (claves, información de pagos).
  • Consentimiento expreso al registrar clientes sobre uso de datos.
  • Política de retención y eliminación de datos cuando ya no sean necesarios.
  • Implementar opciones para que el cliente consulte, modifique o elimine su información conforme a la ley

    • Consecuencias:

    ✅ Positivo: Aumenta la confianza de los usuarios, asegura el cumplimiento de regulaciones nacionales e internacionales y evita sanciones legales.
    ❌ Negativo: Incrementa la complejidad del desarrollo y la gestión documental, además de exigir actualizaciones constantes según cambios en la normativa.



    Decisión Arquitectónica 17: Documentación y manuales de soporte con alertas administrativas

    Contexto:

    El sistema será usado por diferentes perfiles (clientes, recepcionistas, gerentes, administradores) que requieren orientación clara para utilizar correctamente las funciones. Además, el personal de TI y administración necesita recibir alertas sobre el estado del sistema para prevenir fallos.

    Decisión:

    Implementar un esquema de documentación integral y alertas de soporte, que contemple:

  • Manuales de usuario y guías rápidas para cada perfil (con ejemplos de uso de reservas, pagos, PQRS+, etc.).
  • Documentación técnica para desarrolladores y administradores (diagramas, APIs, configuración del servidor).
  • Alertas administrativas automáticas sobre sucesos críticos (ej. espacio de BD cercano al límite, fallos en backup, errores recurrentes en módulos).
  • Un panel de monitoreo para centralizar las alertas y registrar acciones correctivas.

    • Consecuencias:

    ✅ Positivo: Facilita el aprendizaje de usuarios, mejora la gestión del sistema y reduce tiempos de respuesta ante fallos.
    ❌ Negativo: Requiere inversión de tiempo en redacción y mantenimiento de la documentación, además de configurar un sistema de monitoreo.



    Decisión Arquitectónica 18: Compatibilidad con múltiples dispositivos (PC, tablets y smartphones)

    Contexto:

    Los clientes y el personal del hotel acceden al sistema desde diferentes dispositivos: computadores de escritorio en recepción, tablets para gerencia o supervisión, y teléfonos móviles para clientes. Es necesario garantizar que la experiencia de usuario sea consistente y funcional en cualquier pantalla.

    Decisión:

    Diseñar la aplicación con una interfaz web responsiva utilizando estándares como HTML5, CSS3 y JavaScript con frameworks adaptativos (ej. Bootstrap o similares), asegurando que los módulos de reservas, pagos, PQRS+ e informes funcionen correctamente en PC, tablets y smartphones.

    Consecuencias:

    ✅ Positivo: Permite a los clientes y empleados acceder al sistema desde cualquier dispositivo sin instalar aplicaciones adicionales.
    ❌ Negativo: Aumenta el esfuerzo en diseño, pruebas de usabilidad y compatibilidad entre navegadores y resoluciones de pantalla.
