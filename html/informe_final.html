1. Resumen Ejecutivo

El Sistema de Gestión Hotelera del Hotel Cacique Bolívar es una plataforma diseñada para modernizar y optimizar los procesos internos mediante una arquitectura Cliente–Servidor organizada en capas y comunicada a través de APIs REST. El propósito principal es reemplazar los procesos manuales por una solución digital eficiente, escalable y preparada para integraciones futuras.

El sistema unifica la información de clientes, reservas, habitaciones, paquetes turísticos, pagos y PQRS+, garantizando procesos más ágiles y una experiencia digital coherente para los colaboradores y huéspedes del hotel.

Estilos arquitectónicos implementados

  • Arquitectura por capas: Presentación, Aplicación, Dominio e Infraestructura.
  • Cliente–Servidor: interfaz web conectada con backend centralizado.
  • APIs REST: comunicación ligera, estándar y escalable.
  • Orientada a eventos (preparada): apta para futura migración a microservicios.

Módulos principales

  • Gestión de clientes
  • Gestión de reservas
  • Gestión de habitaciones
  • Gestión de paquetes turísticos
  • Gestión de pagos
  • Gestión de PQRS+
  • Consultas e informes

Evidencias disponibles

  • Casos de uso y escenarios
  • Diagramas UML y C4
  • Modelo y diccionario de datos
  • Requisitos funcionales y no funcionales
  • Diseño de interfaces de usuario

Resultado global del sistema

La solución proporciona:

  • Mayor eficiencia operativa
  • Reducción de errores y reprocesos
  • Mejor tiempo de respuesta al cliente
  • Experiencia digital moderna
  • Base sólida para futuras integraciones y microservicios

2. Arquitectura adoptada y justificación

La arquitectura del Sistema de Gestión Hotelera del Hotel Cacique Bolívar fue construida para garantizar escalabilidad, mantenibilidad y separación clara de responsabilidades. Para ello se adoptó una arquitectura por capas, un modelo Cliente–Servidor y comunicación mediante APIs REST.

2.1 Arquitectura por capas

El sistema está estructurado en capas independientes para mejorar el mantenimiento, la claridad del código y la evolución del software.

Capa de Presentación

Contiene las vistas y formularios utilizados por clientes, recepcionistas y administradores. Su objetivo es ofrecer interfaces claras, simples y eficientes.

Capa de Aplicación

Gestiona procesos, validaciones y flujos internos. Aquí se coordina la disponibilidad de habitaciones, creación de reservas y procesamiento general de solicitudes.

Capa de Dominio

Encapsula las reglas fundamentales del negocio hotelero: ciclo de vida de reservas, estados de habitaciones, reglas de pago y gestión de PQRS+.

Capa de Infraestructura

Gestiona el acceso a datos mediante repositorios JPA, consultas y conexiones SQL. Su función es asegurar integridad y persistencia confiable.

Justificación del enfoque por capas

  • Permite agregar nuevas funciones sin afectar módulos existentes.
  • Facilita pruebas, mantenimiento y modularidad.
  • Reduce el acoplamiento y mejora la estructura interna del sistema.

2.2 Cliente–Servidor y uso de APIs REST

El modelo Cliente–Servidor permite que múltiples usuarios operen desde distintos dispositivos mientras el servidor maneja la lógica central y la persistencia.

Ventajas de usar APIs REST

  • Compatibilidad amplia y estándar industrial.
  • Rendimiento rápido y comunicación ligera.
  • Posibilidad de reutilizar backend con apps móviles o kioscos.

2.3 Arquitectura orientada a eventos (preparada)

El sistema está diseñado para soportar eventos en futuras versiones, permitiendo una transición flexible a microservicios.

Eventos futuros posibles

  • ReservaCreada → Actualizar disponibilidad
  • PagoRealizado → Confirmar reservas
  • PQRSRegistrada → Notificar a gerencia

2.4 Preparación para microservicios y bounded contexts

El sistema está dividido conceptualmente en bounded contexts, facilitando una futura migración hacia microservicios sin modificar la estructura central.

  • Clientes
  • Reservas
  • Habitaciones
  • Pagos
  • Paquetes turísticos
  • PQRS+
  • Administración e Informes

Diseño e implementación (resumen)

Arquitectura por capas

El sistema está estructurado bajo una arquitectura por capas, cada una con responsabilidades claramente definidas:

  • Presentación: controladores REST y vistas HTML integradas con Thymeleaf.
  • Aplicación: servicios, validaciones, DTOs, eventos y handlers.
  • Dominio: entidades del negocio, lógica de dominio, factory y observadores.
  • Infraestructura: repositorios propios, notificaciones y componentes técnicos.
  • Persistencia: DAOs internos (no se utiliza Spring Data JPA).

Estructura de carpetas en el repositorio:

  • /HotelCaciqueAS/HotelCaciqueAS/src/main/java/
  • /HotelCaciqueAS/HotelCaciqueAS/src/main/resources/
  • /HotelCaciqueAS/HotelCaciqueAS/target/classes/templates/
  • /HotelCaciqueAS/HotelCaciqueAS/pom.xml

Patrones de diseño aplicados

  • MVC (Modelo–Vista–Controlador): aplicado con controladores REST, entidades y plantillas HTML.
  • Repository: implementado mediante DAOs propios ubicados en la capa de infraestructura.
  • Observer: aplicado para manejar notificaciones internas mediante observadores y sujetos.

Comunicación por eventos

El sistema implementa un modelo básico de eventos internos para ejecutar acciones automáticas cuando se registra una reserva, como enviar notificaciones.

  • ReservaCreadaEvent: evento principal del sistema.
    Productor: ReservaService
    Consumidor: NotificadorEmailHandler

Ruta de eventos:
/HotelCaciqueAS/HotelCaciqueAS/src/main/java/com/hotelcacique/reservas/capa_aplicacion/events/

APIs y contratos

No se utiliza Swagger en este proyecto.

Endpoints principales:

  • POST /reservas — crear reserva.
  • GET /reservas — listar reservas.
  • GET /reservas/{id} — consultar reserva por ID.
  • DELETE /reservas/{id} — eliminar reserva.

Manejo de errores: implementado mediante respuestas controladas en los controladores REST y validaciones internas.

Modularización (bounded contexts)

El sistema está compuesto por un único contexto funcional:

  • Contexto de Reservas: incluye habitaciones, validaciones, creación de reservas, eventos internos y notificaciones.

APIs internas y configuración:

  • /HotelCaciqueAS/HotelCaciqueAS/src/main/resources/application.properties

ADRs incluidos

A continuación se describen los Architecture Decision Records (ADRs) más relevantes que se tomaron durante el desarrollo del Sistema Hotel Cacique Bolívar. Cada ADR documenta el contexto, la decisión tomada y sus consecuencias técnicas.

ADR 1 – Selección de Spring Boot como framework principal

Contexto: Se requería un framework estable, escalable y con integración nativa a Java.

Decisión: Adoptar Spring Boot para gestionar controladores, servicios, repositorios y seguridad.

Consecuencias: Simplifica la configuración, mejora la productividad y ofrece soporte a largo plazo; sin embargo, aumenta el peso del proyecto y requiere mayor conocimiento de Spring.

ADR 2 – Uso del patrón MVC como estilo arquitectónico

Contexto: Se necesitaba separar la lógica, las vistas y el flujo de datos para asegurar mantenibilidad.

Decisión: Implementar el sistema bajo el patrón MVC con arquitectura por capas.

Consecuencias: Permite evolución modular y facilita pruebas; aumenta la complejidad al inicio por la separación estricta.

ADR 3 – Base de datos relacional MySQL

Contexto: El sistema exige integridad, consistencia y transacciones seguras.

Decisión: Seleccionar MySQL como motor de base de datos principal.

Consecuencias: Garantiza integridad y amplia documentación, pero puede limitar la escalabilidad horizontal en escenarios futuros.

ADR 4 – Relación uno a uno entre Pagos y Reservas

Contexto: Se buscaba evitar pagos duplicados o inconsistentes en una misma reserva.

Decisión: Implementar una relación estricta 1:1 entre pago y reserva.

Consecuencias: Mejora la trazabilidad del proceso de pago, pero obliga a validaciones adicionales al eliminar o modificar una reserva.

ADR 5 – Implementación del módulo PQRS+ basado en roles

Contexto: Los roles del sistema (cliente, recepcionista, administrador) tenían diferentes permisos.

Decisión: El cliente puede CRUD completo; la recepcionista solo puede consultar.

Consecuencias: Aumenta la seguridad y evita modificaciones indebidas; implica lógica de permisos más compleja.

ADR 6 – Integración de validaciones de archivos según tipo

Contexto: Se manejaban recursos de tipo imagen y documentos PDF.

Decisión: Implementar validación dual: imágenes (png, jpg, webp) o PDF según elección del usuario.

Consecuencias: Mayor seguridad y control; requiere manejo de errores más detallado en backend.

ADR 7 – Manejo modular de habitaciones y paquetes

Contexto: Los paquetes pueden incluir habitaciones por defecto y habitaciones adicionales.

Decisión: Separar lógica en las tablas Paquete, Paquete_Habitacion y Extras_Paquete.

Consecuencias: Flexibilidad total en la creación de paquetes; complejidad adicional en consultas SQL.

ADR 8 – Validación de disponibilidad de habitaciones desde la capa de servicio

Contexto: Se necesitaba evitar sobre-reservas o conflictos de fechas.

Decisión: Centralizar la validación de disponibilidad en la capa de servicios.

Consecuencias: Garantiza consistencia global; incrementa carga lógica en los servicios.

ADR 9 – Uso de Bootstrap 5 para responsividad

Contexto: El sistema debe funcionar en PC, tablets y móviles.

Decisión: Implementar la interfaz con Bootstrap 5.

Consecuencias: Alta compatibilidad y diseño consistente; limita ciertos estilos sin personalización adicional.

ADR 10 – Implementación de módulos de informes con filtros y exportación

Contexto: Usuarios administrativos necesitan analizar reservas, pagos y clientes.

Decisión: Crear un módulo de informes con filtros y posibilidad de exportar a Excel o PDF.

Consecuencias: Mejora la toma de decisiones; requiere mayor procesamiento en consultas y generación de archivos.

Atributos de calidad (NFRs) y tácticas

El sistema del Hotel Cacique Bolívar fue diseñado considerando requisitos no funcionales que garantizan estabilidad, seguridad, mantenimiento y crecimiento sostenido. A continuación, se presentan los atributos de calidad clave y las tácticas aplicadas.

  • Mantenibilidad
    Tácticas aplicadas:
    • Arquitectura en capas (Presentación, Aplicación, Dominio, Persistencia).
    • Inversión de dependencias (DI) con Spring Boot para desacoplar módulos.
    • Uso de DTOs para separar la vista del modelo interno.
    • Controladores, servicios y repositorios claramente separados según responsabilidad.
    • Estructura modular por dominios (reservas, clientes, PQRS+, pagos, habitaciones).
  • Escalabilidad
    Tácticas aplicadas:
    • Servicios sin estado (stateless) en la capa de aplicación.
    • Uso de eventos internos con Observer para desacoplar operaciones.
    • Posibilidad futura de escalado horizontal en contenedores.
    • Base de datos optimizada con índices para consultas masivas.
  • Disponibilidad
    Tácticas aplicadas:
    • Manejo centralizado de errores en controladores.
    • Validación de entradas y protección ante datos corruptos.
    • Posibilidad de health checks mediante Actuator (integrable con Spring Boot).
    • Backups periódicos de la base de datos (definidos en ADR 8).
  • Desempeño
    Tácticas aplicadas:
    • Paginación y filtrado en consultas de reservas, clientes y PQRS+.
    • Índices en campos críticos (documento, códigos de reserva, habitaciones).
    • Optimización de consultas repetitivas.
    • Uso futuro de caché para disponibilidad de habitaciones (propuesto en ADR 14).
  • Seguridad
    Tácticas aplicadas:
    • Roles y permisos definidos (administrador, recepcionista, cliente, gerencia).
    • Validaciones en formularios y cargas de documentos.
    • HTTPS/TLS para proteger la comunicación (ADR 006).
    • Gestión segura de credenciales en el servidor.
    • Restricción de acceso a datos sensibles (pagos, PQRS+).
  • Observabilidad
    Tácticas aplicadas:
    • Logs estructurados en operaciones críticas (reservas, pagos, PQRS+).
    • Registros de errores explícitos para facilitar diagnóstico.
    • Métricas internas y posibilidad de integración con Actuator.
    • Trazabilidad mediante eventos internos (Observer).

Conclusiones

Estilos arquitectónicos implementados

El sistema fue construido sobre una arquitectura en capas combinada con un modelo MVC y soporte para eventos internos (Observer). Esta elección permitió separar de forma clara la presentación, la lógica de aplicación, el dominio y la persistencia, facilitando la evolución del software y reduciendo el acoplamiento.

Beneficios obtenidos

  • Mantenibilidad: gracias a capas bien definidas, uso de servicios, DTOs, repositorios y una estructura modular fácil de extender.
  • Escalabilidad: el sistema está preparado para crecer mediante eventos, integración futura con microservicios de IA, y separación clara de responsabilidades.
  • Desacoplamiento: el patrón MVC, el uso de interfaces y la organización por dominios reducen dependencias directas y permiten intercambiar componentes sin afectar el sistema completo.
  • Confiabilidad: la integración con MySQL, validaciones internas, logs y manejo estructurado de errores garantizan una operación estable.

Próximos pasos recomendados

  • Integrar un microservicio de analítica e IA para predicción de ocupación y recomendaciones inteligentes de paquetes.
  • Implementar un sistema avanzado de seguridad con autenticación JWT/OAuth2 y cifrado integral para la protección de datos personales y financieros.
